# video_tool

`video_tool` 是我做的一款 Windows 桌面视频工具，主要用来把常见的视频保存、音频提取、批量任务管理和历史记录整理做得更顺手。它的重点不是让用户面对复杂参数，而是把日常会用到的操作放进一个清楚、稳定、可以直接点击使用的桌面应用里。

[前往 Releases 下载最新版](https://github.com/dec-777/video_tool/releases/latest)

## 项目定位

我做这个项目的出发点很简单：平时保存视频、提取音频、整理素材时，经常需要重复复制链接、选择目录、等待任务、查找下载结果。`video_tool` 把这些流程整理成一个桌面端工作台，让下载、排队、重试、打开文件夹和查看历史都集中在同一个界面里完成。

适合的使用场景：

- 保存单个视频到本地。
- 从视频中提取 MP3 或 M4A 音频。
- 一次性处理多个链接。
- 管理正在下载、已完成、失败和取消的任务。
- 通过历史记录找回之前处理过的内容。
- 给下载文件统一设置保存目录和命名规则。

## 功能介绍

### 单链接处理

- 输入一个视频链接后，可以先解析视频信息。
- 解析完成后可以查看标题、来源、封面等基础信息。
- 支持下载 MP4 视频。
- 支持提取 MP3 / M4A 音频。
- 支持自定义保存目录。
- 支持自定义文件名模板，方便整理文件。
- 链接无法解析或下载失败时，会显示可读的错误提示。

### 批量任务

- 支持一次粘贴多个链接。
- 每行一个链接，空行会自动忽略。
- 重复链接会自动去重。
- 每个链接都会生成独立任务。
- 单个任务失败不会影响其他任务继续执行。
- 适合一次性处理课程、素材、收藏夹整理等批量场景。

### 下载队列

- 支持任务排队下载。
- 可以查看任务状态、进度、速度和剩余时间。
- 支持取消正在进行的任务。
- 失败任务可以重新尝试。
- 下载完成后可以直接打开文件所在文件夹。
- 每个任务都有独立状态，方便判断当前处理进度。

### 历史记录

- 自动记录完成、失败等任务结果。
- 可以查看标题、链接、保存位置、完成时间和错误信息。
- 支持从历史记录重新发起下载。
- 支持清空历史记录。
- 清空历史只会清除应用里的记录，不会删除已经下载到本地的文件。

### 字幕与附加内容

- 支持在可用时提取字幕。
- 支持选择字幕格式。
- 支持字幕单独保存或配合视频任务使用。
- 支持按实际链接可提供的内容进行处理，不强行生成不存在的资源。

### 设置

- 默认保存目录。
- 默认下载模式。
- 默认音频格式。
- 默认文件名模板。
- 并发任务数量。
- 本地运行环境检测。
- 恢复默认设置。

## 使用流程

1. 打开 `video_tool`。
2. 在首页粘贴视频链接。
3. 点击解析，确认视频信息。
4. 选择下载类型，例如 MP4 视频、MP3 音频或 M4A 音频。
5. 选择保存目录，也可以使用默认目录。
6. 点击开始下载。
7. 在任务页面查看进度、速度、剩余时间和任务结果。
8. 下载完成后打开文件夹查看文件。

批量下载时，进入批量页面，把多个链接按一行一个的方式粘贴进去，再统一加入任务即可。

## 本地运行源码

如果你想在本地运行源码，需要先安装：

- Windows 10 / Windows 11
- Node.js 20+
- npm
- PowerShell 5+

运行步骤：

```powershell
git clone https://github.com/dec-777/video_tool.git
cd video_tool
npm install
npm run setup:binaries
npm run dev
```

首次运行源码前，`npm run setup:binaries` 会准备本地运行需要的组件。仓库默认不提交这些大文件，所以新电脑部署时需要先执行这一步。

## 常见问题

### npm run dev 应该在哪里运行？

需要在项目根目录运行，也就是包含 `package.json` 的目录：

```powershell
cd E:\du\AI\ai_coding\codex\video_project_codex
npm run dev
```

如果在上一级目录运行，会因为找不到 `package.json` 而启动失败。

### 端口被占用怎么办？

开发模式会使用本项目自己的本地端口。如果提示端口被占用，通常是之前的开发服务还没有关闭。关闭旧的终端窗口，或者在任务管理器里结束残留的 Node / Electron 进程后，再重新运行：

```powershell
npm run dev
```

### 为什么仓库里没有提交大体积运行文件？

这些文件体积比较大，不适合直接放进 Git 仓库。项目提供了准备脚本，新环境中执行一次即可。
