# yt-dlp Windows 桌面下载管理器项目架构文档

> 文档版本：V1.0  
> 项目类型：Windows 桌面程序  
> 技术栈：Electron + React + Vite + Node.js + yt-dlp + FFmpeg  
> 目标平台：Windows 10 / Windows 11  
> 文档目标：明确项目整体架构、目录结构、模块职责、数据流、事件流、依赖边界和可扩展设计，指导后续开发和维护。

---

## 1. 架构设计目标

本项目的核心目标是将 yt-dlp 的命令行能力封装为一个可视化、稳定、安全、可扩展的 Windows 桌面下载管理器。

架构设计需要满足以下目标：

1. 前端 UI 与本地后端逻辑分离。
2. React 只负责界面展示和用户交互。
3. Electron Main Process 负责系统能力和本地后端能力。
4. 所有 yt-dlp 参数统一由 commandBuilder 生成。
5. 所有下载任务统一由 taskQueue 管理。
6. 所有进程调用统一由 ytdlpService 管理。
7. 所有配置、历史、日志统一由 Store / Service 管理。
8. 不允许前端直接调用系统命令。
9. 支持 Windows 中文路径、空格路径和打包路径。
10. 支持后续扩展播放列表、Cookie、代理、字幕、封面、元数据、自动更新等能力。

### 2.1 总体架构

```text
┌──────────────────────────────────────────────┐
│                 用户操作层                    │

                      │

┌──────────────────────────────────────────────┐
│                React 渲染层                   │
│  Pages / Components / Stores / Hooks          │
└─────────────────────┬────────────────────────┘
                      │
                      │ window.api.xxx()
                      ↓
┌──────────────────────────────────────────────┐
│              Preload 安全桥接层               │
│       contextBridge 暴露白名单 API             │
└─────────────────────┬────────────────────────┘
                      │
                      │ IPC invoke / on
                      ↓
┌──────────────────────────────────────────────┐
│          Electron Main Process 本地后端层      │
│ IPC Router / Services / Queue / Stores        │
└─────────────────────┬────────────────────────┘
                      │
                      │ spawn(args[])
                      ↓

│              外部二进制执行层                 │

                      │
                      ↓
┌──────────────────────────────────────────────┐
│                本地文件系统                   │
│  下载文件 / 配置 / 历史 / 日志 / archive       │
└──────────────────────────────────────────────┘
```

### 2.2 架构分层

项目分为 6 层：

```text
1. UI 展示层
2. 前端状态层
3. IPC 通信层
4. 本地后端服务层
5. 任务执行层
6. 本地持久化层
```

| 层级 | 主要目录 | 主要职责 |
|---|---|---|
| UI 展示层 | `src/pages`、`src/components` | 页面展示、用户操作、表单输入 |
| 前端状态层 | `src/store`、`src/hooks` | 管理前端任务状态、配置状态、UI 状态 |
| IPC 通信层 | `electron/preload.js`、`electron/ipc` | 前后端安全通信 |
| 本地后端服务层 | `electron/services` | yt-dlp 调用、命令生成、文件处理、配置历史日志 |
| 任务执行层 | `electron/services/taskQueue.js`、`ytdlpService.js` | 下载队列、进程管理、进度解析 |
| 本地持久化层 | `userData`、`data`、`logs` | 配置、任务、历史、日志、archive |

## 4. 目录职责说明

---

## 4.1 `electron/`

该目录存放 Electron 主进程相关代码，也就是本地后端代码。

### 职责

1. 创建桌面窗口。
2. 注册 IPC 接口。
3. 调用 yt-dlp 和 FFmpeg。
4. 管理下载任务。
5. 读写配置和历史。
6. 处理本地文件系统。
7. 写入日志。
8. 处理打包路径和资源路径。

---

## 4.2 `electron/ipc/`

该目录存放 IPC 注册逻辑。

### 设计原则

1. 按功能拆分 IPC。
2. `main.js` 不直接写大量 `ipcMain.handle`。
3. 每个 IPC 文件只负责一个领域。
4. IPC 层只做参数接收、校验和调用 service。
5. 业务逻辑不写在 IPC 层。

### 示例结构

```text
electron/ipc/
├── index.js
├── downloadIpc.js
├── configIpc.js
├── fileIpc.js
├── historyIpc.js
└── systemIpc.js
```

### 职责划分

| 文件 | 职责 |
|---|---|
| `index.js` | 统一注册所有 IPC |
| `downloadIpc.js` | 下载、解析、任务相关 IPC |
| `configIpc.js` | 配置读取、保存 |
| `fileIpc.js` | 选择文件、选择目录、打开文件夹 |
| `historyIpc.js` | 下载历史读取、清空、删除 |
| `systemIpc.js` | 检查版本、检查二进制文件 |

---

该目录存放本地后端核心服务。

## 4.6 `src/components/`

组件级 UI。

设计原则：

1. 组件只负责局部 UI。
2. 组件尽量无业务副作用。
3. 复杂业务放在 hooks 或 store。
4. 通用组件和业务组件分开。

---

## 4.7 `src/store/`

前端状态管理。

推荐使用 Zustand。  
原因：

1. 简单。
2. 文件结构清晰。
3. 不需要大量样板代码。
4. 适合 Electron 工具类项目。

### Store 划分

| Store | 职责 |
|---|---|
| `taskStore.js` | 前端任务列表、进度、状态 |
| `configStore.js` | 前端配置缓存 |
| `historyStore.js` | 历史记录缓存 |
| `uiStore.js` | 主题、侧边栏、弹窗、加载状态 |

---

## 4.8 `src/api/`

封装前端对 `window.api` 的调用。

目的：

1. 避免页面直接到处写 `window.api.xxx`。
2. 统一处理错误。
3. 方便后续测试和替换。
4. 让 UI 代码更干净。

示例：

```js
export async function parseUrl(url) {
  return window.api.parseUrl(url);
}

export async function startDownload(options) {
  return window.api.startDownload(options);
}
```

## 6. 核心服务架构

---

## 6.1 `commandBuilder.js`

### 定位

命令生成中心。

### 原则

1. 前端不能直接传 yt-dlp 参数数组。
2. 前端只能传结构化配置。
3. commandBuilder 根据白名单生成参数。
4. commandBuilder 返回 `args[]`。
5. commandBuilder 不执行命令。

### 输入

```js
DownloadOptions
```

### 输出

```js
[
  "--newline",
  "--no-color",
  "--ffmpeg-location",
  "D:/app/bin",
  "-f",
  "bv*+ba/b",
  "--merge-output-format",
  "mp4",
  "-o",
  "D:/Downloads/%(title)s.%(ext)s",
  "https://example.com/video"
]
```

### 内部结构

```text
commandBuilder.js
├── buildYtdlpArgs()
├── buildBaseArgs()
├── buildFormatArgs()
├── buildSubtitleArgs()
├── buildMetadataArgs()
├── buildAuthArgs()
├── buildNetworkArgs()
├── buildFileArgs()
├── buildPlaylistArgs()
├── buildCommandPreview()
└── validateDownloadOptions()
```

---

## 6.2 `ytdlpService.js`

### 定位

yt-dlp 进程执行中心。

### 只负责

1. 启动 yt-dlp。
2. 读取 stdout。
3. 读取 stderr。
4. 返回结果。
5. 抛出错误。
6. 提供取消进程能力。

### 不负责

1. 不生成复杂参数。
2. 不管理队列。
3. 不维护 UI 状态。
4. 不直接写 React store。

### 方法设计

```js
async function getVideoInfo(url, runtimeOptions)
async function getPlaylistInfo(url, runtimeOptions)
function startDownloadProcess(task, args, callbacks)
function killProcess(taskId)
async function checkVersion()
async function updateYtdlp()
```

---

## 6.3 `taskQueue.js`

### 定位

下载任务调度中心。

### 职责

1. 接收任务。
2. 排队。
3. 控制并发。
4. 启动下载。
5. 更新状态。
6. 取消任务。
7. 重试任务。
8. 推送事件。
9. 持久化任务。
10. 完成后写入历史。

### 内部数据

```js
const tasks = new Map();
const pendingQueue = [];
const runningTasks = new Map();
```

### 关键方法

```js
addTask(options)
addBatchTasks(optionsList)
startNext()
cancelTask(taskId)
retryTask(taskId)
updateTask(taskId, patch)
getTasks()
pauseQueue()
resumeQueue()
```

---

## 6.4 `progressParser.js`

### 定位

yt-dlp 输出解析器。

### 输入

```text
[download]  35.2% of 50.00MiB at 2.30MiB/s ETA 00:15
```

### 输出

```js
{
  percent: 35.2,
  totalSize: "50.00MiB",
  speed: "2.30MiB/s",
  eta: "00:15",
  status: "downloading"
}
```

### 需要识别的阶段

```text
[download]       downloading
[Merger]         merging
[ExtractAudio]   postprocessing
[Metadata]       postprocessing
[EmbedSubtitle]  postprocessing
ERROR            failed
```

---

## 6.5 `configStore.js`

### 定位

配置持久化服务。

### 要求

1. 使用 `app.getPath("userData")`。
2. 配置不存在时创建默认配置。
3. 配置损坏时备份并重建。
4. 保存时使用原子写入。
5. 支持配置版本迁移。

### 文件位置

```text
C:/Users/用户名/AppData/Roaming/YT-DLP Desktop/config.json
```

---

## 6.6 `historyStore.js`

### 定位

下载历史持久化服务。

### 要求

1. 任务完成后写入。
2. 任务失败后写入。
3. 支持读取全部历史。
4. 支持搜索。
5. 支持删除单条。
6. 支持清空。
7. 不和 archive 混用。

---

## 6.7 `logService.js`

### 定位

日志服务。

### 日志类型

```text
app.log
task-{taskId}.log
```

### 要求

1. 不记录 Cookie 内容。
2. 错误必须记录原始摘要。
3. 任务日志可以由 UI 查看。
4. 日志可以导出。
5. 日志文件过大时需要轮转，V1 可以先限制最大行数或最大大小。

## 12. 状态机设计

---

## 12.1 任务状态机

```text
┌─────────┐
│ pending │
└────┬────┘
     │ start
     ↓
┌─────────┐
│ parsing │
└────┬────┘
     │ parsed
     ↓
┌─────────────┐
│ downloading │
└────┬────────┘
     │ merger / postprocess
     ↓
┌─────────┐
│ merging │
└────┬────┘
     │ postprocess
     ↓
┌────────────────┐
│ postprocessing │
└────┬───────────┘
     │ success
     ↓
┌───────────┐
│ completed │
└───────────┘
```

异常流：

```text
pending / parsing / downloading / merging / postprocessing
 ↓
failed
```

取消流：

```text
pending / parsing / downloading
 ↓
canceled
```

---

## 12.2 状态流转规则

| 当前状态 | 可变为 | 说明 |
|---|---|---|
| pending | parsing / downloading / canceled | 等待中可启动或取消 |
| parsing | downloading / failed / canceled | 解析中可成功、失败、取消 |
| downloading | merging / postprocessing / completed / failed / canceled | 下载中可能进入后处理 |
| merging | postprocessing / completed / failed | 合并后完成或继续后处理 |
| postprocessing | completed / failed | 后处理结束 |
| completed | pending | 可重新下载 |
| failed | pending | 可重试 |
| canceled | pending | 可重新加入队列 |

## 15. 安全架构

---

## 15.1 前端安全边界

React 前端不能访问：

1. Node.js API。
2. 文件系统 API。
3. child_process。
4. shell 命令。
5. yt-dlp.exe。
6. ffmpeg.exe。

React 只能访问：

```js
window.api
```

---

## 15.2 Preload 安全边界

preload 只暴露有限 API。

允许：

```js
window.api.parseUrl()
window.api.startDownload()
window.api.cancelTask()
window.api.selectFolder()
window.api.openFolder()
```

禁止：

```js
window.api.runCommand()
window.api.exec()
window.api.eval()
window.api.getNodeApi()
```

---

## 15.3 命令生成安全边界

用户输入只允许进入：

1. URL。
2. 文件路径。
3. 代理地址。
4. 文件名模板。
5. 字幕语言。
6. 格式 ID。

所有输入必须校验。

不允许用户直接输入完整命令并执行。

---

## 15.4 Cookie 安全

1. Cookie 内容不展示。
2. Cookie 内容不写入日志。
3. Cookie 文件路径可以保存。
4. 浏览器 Cookie 只作为 yt-dlp 参数使用。
5. 命令预览不显示 Cookie 内容。
6. UI 明确提示 Cookie 是敏感登录凭证。

## 23. 架构约束

开发时必须遵守以下约束：

1. React 页面不得直接调用 Node.js。
2. React 页面不得直接访问 `ipcRenderer`。
3. 前端不得传入任意命令字符串。
4. 所有 yt-dlp 参数必须由 commandBuilder 生成。
5. 所有进程启动必须通过 ytdlpService。
6. 所有任务状态必须通过 taskQueue 更新。
7. 所有配置必须通过 configStore 读写。
8. 所有历史必须通过 historyStore 读写。
9. 所有日志必须通过 logService 写入。
10. 所有路径必须通过 pathUtils / fileService 处理。
11. 所有 IPC 名称必须使用常量。
12. 所有错误必须经过 errorService 归一化。
13. 所有下载进度必须通过 progressParser 解析。
14. 不允许在 main.js 中堆积大量业务逻辑。
15. 不允许在 React 页面中堆积复杂下载逻辑。

## 24. 架构总结

本项目采用“前端 UI + Electron 本地后端 + yt-dlp 执行核心”的架构。

最核心的 5 个模块是：

```text
1. commandBuilder.js
   负责把 UI 配置转换成 yt-dlp 参数。

2. ytdlpService.js
   负责执行 yt-dlp 子进程。

3. taskQueue.js
   负责下载队列、任务状态、并发控制。

4. progressParser.js
   负责解析 yt-dlp 输出并生成进度状态。

5. configStore / historyStore / logService
   负责配置、历史和日志持久化。
```

最核心的架构原则是：

```text
React 不碰系统能力；
preload 只暴露安全 API；
IPC 只做通信；
service 负责业务；
yt-dlp 只通过 spawn 参数数组执行；
配置、历史、日志全部本地持久化；
高级功能按 V1 / V2 / V3 分阶段扩展。
```

只要严格遵守该架构，项目就可以在第一版保持可控，并且为后续扩展播放列表、Cookie、代理、字幕、元数据、自动更新、SQLite、浏览器插件等功能保留足够空间。
