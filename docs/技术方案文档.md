# yt-dlp Windows 桌面下载管理器技术方案

> 文档版本：V1.0  
> 对应需求文档：yt-dlp Windows 桌面下载管理器需求文档 V1.0  
> 项目类型：Windows 桌面程序  
> 技术栈：Electron + React + Vite + Node.js + yt-dlp + FFmpeg  
> 目标平台：Windows 10 / Windows 11  
> 文档目标：指导开发者按照合理工程架构完成项目开发、调试、打包和后续扩展

---

## 1. 技术方案概述

本项目采用 **Electron + React + yt-dlp.exe + FFmpeg.exe** 的技术路线。

整体思路是：

```text
React 负责桌面 UI
Electron Main Process 负责本地后端能力
preload.js 负责安全桥接
IPC 负责前后端通信
yt-dlp.exe 负责解析和下载
FFmpeg 负责音视频合并、转码、字幕/封面/元数据处理
本地 JSON 或 SQLite 负责配置、任务、历史记录持久化
```

本项目不修改 yt-dlp 源码，而是将 yt-dlp 作为本地下载核心，通过可视化 UI 生成 yt-dlp 参数，并用 Node.js `child_process.spawn` 启动 yt-dlp 子进程完成下载。

## 2. 核心技术选型

---

## 2.1 桌面框架：Electron

### 选择原因

1. 支持 Windows 桌面程序开发。
2. 可以使用 React/Vue 等前端技术构建现代 UI。
3. Node.js 主进程可以调用本地 exe 文件。
4. 打包 Windows 安装包成熟。
5. 社区成熟，资料丰富。
6. 适合快速开发可视化工具类桌面软件。

### Electron 在本项目中的作用

1. 创建 Windows 桌面窗口。
2. 管理主进程和渲染进程。
3. 通过 IPC 连接 UI 和本地后端。
4. 调用 yt-dlp.exe 和 ffmpeg.exe。
5. 访问本地文件系统。
6. 选择文件夹、打开文件、打开目录。
7. 打包成 Windows 安装包。

---

## 2.2 前端框架：React

### 选择原因

1. 组件化清晰。
2. 适合构建复杂工具型 UI。
3. 状态管理成熟。
4. 适合做任务列表、设置页、表单页、命令预览等界面。
5. 和 Vite / Electron 集成方便。

### React 在本项目中的作用

1. 首页下载界面。
2. 批量 URL 输入界面。
3. 播放列表解析界面。
4. 格式与清晰度选择界面。
5. 字幕、封面、元数据设置界面。
6. Cookie、代理设置界面。
7. 文件管理界面。
8. 下载任务中心。
9. 下载历史。
10. 命令预览。

---

## 2.3 构建工具：Vite

### 选择原因

1. 开发启动快。
2. React 支持成熟。
3. 配置简单。
4. 适合 Electron 渲染进程开发。
5. 打包输出结构清晰。

---

## 2.4 本地后端：Electron Main Process + Node.js

### 选择原因

Electron 主进程天然具备 Node.js 能力，可以：

1. 调用本地程序。
2. 读写文件。
3. 管理子进程。
4. 处理系统路径。
5. 管理下载队列。
6. 写入日志和历史记录。

本项目不需要单独启动 HTTP 后端服务器。  
原因：

1. 桌面程序本地运行，不需要网络 API。
2. Electron IPC 足以完成前后端通信。
3. 减少部署复杂度。
4. 避免端口占用问题。
5. 降低安全风险。

---

## 2.5 下载核心：yt-dlp.exe

### 作用

1. 解析视频网页。
2. 获取视频标题、封面、时长、格式列表。
3. 下载视频。
4. 下载音频。
5. 下载字幕。
6. 下载封面。
7. 下载元数据。
8. 使用 Cookie。
9. 使用代理。
10. 处理播放列表、频道、合集等。

### 本项目调用方式

使用 Node.js：

```js
child_process.spawn(ytdlpPath, args)
```

禁止使用：

```js
child_process.exec(commandString)
```

原因：

1. `spawn` 使用参数数组，更适合处理路径空格和中文。
2. `spawn` 不需要拼接命令字符串，安全性更高。
3. `spawn` 可以实时监听 stdout 和 stderr。
4. `exec` 有输出缓冲限制，不适合长时间下载任务。
5. `exec` 更容易出现命令注入风险。

## 2.6 音视频处理：FFmpeg / FFprobe

### 作用

1. 合并视频流和音频流。
2. 转换音频格式。
3. 转换字幕格式。
4. 嵌入字幕。
5. 嵌入缩略图。
6. 嵌入元数据。
7. 处理后期格式转换。

### 文件要求

程序需要内置：

```text
bin/ffmpeg.exe
bin/ffprobe.exe
```

yt-dlp 调用时统一添加：

```bash
--ffmpeg-location "bin目录"
```

## 2.7 数据存储方案

### V1 推荐：JSON 文件

V1 阶段建议使用 JSON 文件保存配置、任务、历史记录。

原因：

1. 实现简单。
2. 调试方便。
3. 不需要数据库依赖。
4. 适合 MVP。
5. 数据量不大时性能足够。

文件结构：

```text
data/config.json
data/tasks.json
data/history.json
data/archive.txt
logs/app.log
```

### V2 可升级：SQLite

当下载历史、任务记录、搜索筛选变复杂时，可以升级 SQLite。

适合场景：

1. 历史记录超过数千条。
2. 需要复杂搜索。
3. 需要按站点、时间、状态筛选。
4. 需要更可靠的数据一致性。

## 4. 进程模型设计

---

## 4.1 主进程 Main Process

主进程负责所有本地敏感操作。

职责：

1. 创建窗口。
2. 注册 IPC。
3. 调用 yt-dlp。
4. 管理任务队列。
5. 读写配置。
6. 读写历史记录。
7. 打开文件夹。
8. 写入日志。
9. 检查本地二进制文件。
10. 处理打包路径。

主进程文件：

```text
electron/main.js
```

---

## 4.2 渲染进程 Renderer Process

渲染进程是 React 页面。

职责：

1. 展示 UI。
2. 收集用户输入。
3. 展示任务状态。
4. 展示错误信息。
5. 展示命令预览。
6. 通过 `window.api` 调用后端能力。

渲染进程禁止：

1. 直接访问 Node.js API。
2. 直接调用 child_process。
3. 直接读写敏感文件。
4. 直接执行系统命令。

---

## 4.3 Preload 安全桥

`preload.js` 是唯一允许连接主进程和前端的桥。

职责：

1. 使用 contextBridge 暴露有限 API。
2. 不暴露 ipcRenderer 原始对象。
3. 对外提供明确功能方法。
4. 只允许白名单事件订阅。

示例：

```js
const { contextBridge, ipcRenderer } = require("electron");

contextBridge.exposeInMainWorld("api", {
  parseUrl: (url) => ipcRenderer.invoke("parse-url", url),
  startDownload: (options) => ipcRenderer.invoke("start-download", options),
  cancelTask: (taskId) => ipcRenderer.invoke("cancel-task", taskId),
  retryTask: (taskId) => ipcRenderer.invoke("retry-task", taskId),
  getTasks: () => ipcRenderer.invoke("get-tasks"),
  getHistory: () => ipcRenderer.invoke("get-history"),
  getConfig: () => ipcRenderer.invoke("get-config"),
  saveConfig: (config) => ipcRenderer.invoke("save-config", config),
  selectFolder: () => ipcRenderer.invoke("select-folder"),
  selectFile: () => ipcRenderer.invoke("select-file"),
  openFolder: (folderPath) => ipcRenderer.invoke("open-folder", folderPath),

  onTaskProgress: (callback) => {
    ipcRenderer.on("task-progress", (_, data) => callback(data));
  },

  onTaskCompleted: (callback) => {
    ipcRenderer.on("task-completed", (_, data) => callback(data));
  },

  onTaskFailed: (callback) => {
    ipcRenderer.on("task-failed", (_, data) => callback(data));
  }
});
```

## 5.2 ytdlpService.js

### 作用

封装所有与 yt-dlp 相关的操作。

### 主要方法

```js
getVideoInfo(url, options)
getPlaylistInfo(url, options)
startDownload(task, options)
cancelDownload(taskId)
checkYtdlpVersion()
updateYtdlp()
getSupportedSites()
```

### getVideoInfo 实现思路

调用：

```bash
yt-dlp.exe -J "URL"
```

为了避免下载播放列表时一次返回大量数据，解析单视频时可以加：

```bash
--no-playlist
```

示例：

```js
function getVideoInfo(url) {
  const args = ["-J", "--no-playlist", url];
  return runYtdlpJson(args);
}
```

### getPlaylistInfo 实现思路

调用：

```bash
yt-dlp.exe -J --flat-playlist "URL"
```

用于快速获取播放列表条目，不下载完整格式信息。

示例：

```js
function getPlaylistInfo(url) {
  const args = ["-J", "--flat-playlist", url];
  return runYtdlpJson(args);
}
```

### startDownload 实现思路

1. 接收任务对象。
2. 使用 commandBuilder 生成 args。
3. spawn 启动 yt-dlp。
4. 保存 childProcess。
5. 监听 stdout 和 stderr。
6. 解析进度。
7. 更新 taskQueue 状态。
8. 推送 IPC。
9. 任务结束后写入历史。

## 5.3 commandBuilder.js

### 作用

将 UI 配置转换为 yt-dlp 参数数组。

这是整个项目最关键的模块之一。

### 输入

```js
DownloadOptions
```

### 输出

```js
string[]
```

### 设计原则

1. 只生成参数数组，不执行命令。
2. 不拼接 shell 字符串。
3. 不允许任意用户输入变成危险参数。
4. 对路径、URL、格式进行校验。
5. 选项必须有明确来源。
6. 支持命令预览复用。

### 示例

```js
function buildYtdlpArgs(options, runtime) {
  const args = [];

  args.push("--newline");
  args.push("--no-color");

  args.push("--ffmpeg-location", runtime.ffmpegDir);

  buildFormatArgs(args, options);
  buildSubtitleArgs(args, options);
  buildMetadataArgs(args, options);
  buildAuthArgs(args, options);
  buildNetworkArgs(args, options);
  buildFileArgs(args, options);
  buildPlaylistArgs(args, options);

  args.push(...options.urls);

  return args;
}
```

## 5.4 taskQueue.js

### 作用

管理所有下载任务。

### 职责

1. 创建任务。
2. 排队任务。
3. 启动任务。
4. 控制并发数。
5. 暂停队列。
6. 继续队列。
7. 取消任务。
8. 重试任务。
9. 更新任务状态。
10. 推送任务事件。

### 并发策略

V1 默认并发数：

```text
1
```

V2 可配置：

```text
1 - 5
```

不建议默认过高。  
原因：

1. 多站点可能限流。
2. 容易触发 403。
3. 占用带宽。
4. 容易导致 FFmpeg 多进程占用过高。
5. 用户体验不稳定。

### 任务状态机

```text
pending
  ↓
parsing
  ↓
downloading
  ↓
merging
  ↓
postprocessing
  ↓
completed
```

异常路径：

```text
pending / parsing / downloading / merging / postprocessing
  ↓
failed
```

取消路径：

```text
pending / parsing / downloading
  ↓
canceled
```

## 5.5 configStore.js

### 作用

读写用户配置。

### 存储路径

开发环境：

```text

```

生产环境建议：

```text
app.getPath("userData")/config.json
```

原因：

1. 安装目录可能没有写权限。

```json
{
  "defaultOutputDir": "",

  "defaultContainer": "mp4",
  "defaultAudioFormat": "mp3",
  "defaultSubtitleFormat": "srt",
  "proxyEnabled": false,
  "proxyUrl": "",
  "cookiesMode": "none",
  "cookiesFile": "",
  "cookiesFromBrowser": "",
  "archiveEnabled": true,
  "archivePath": "",
  "keepIntermediateFiles": false,
  "concurrentDownloads": 1,
  "downloadThumbnail": false,
  "downloadDescription": false,
  "downloadInfoJson": false,
  "embedMetadata": false,
```

### 要求

1. 配置文件不存在时创建默认配置。
2. 配置文件损坏时备份并恢复默认配置。
3. 保存配置时使用原子写入。
4. 避免写入敏感 Cookie 内容。


## 9.3 取消任务

### pending 状态

```js
child.kill("SIGTERM");
```

## 10. 下载进度解析设计

---

## 10.1 yt-dlp 输出示例

```text
[download]  35.2% of 50.00MiB at 2.30MiB/s ETA 00:15
[Merger] Merging formats into "xxx.mp4"
[ExtractAudio] Destination: xxx.mp3
[Metadata] Adding metadata to "xxx.mp4"
```

---

## 10.2 需要解析的状态

| 输出内容 | 状态 |
|---|---|
| `[download]` | downloading |
| `[Merger]` | merging |
| `[ExtractAudio]` | postprocessing |
| `[Metadata]` | postprocessing |
| `[EmbedSubtitle]` | postprocessing |
| `has already been downloaded` | completed |
| `ERROR:` | failed |

---

## 10.3 进度解析函数

```js
function parseProgressLine(line) {
  const progressRegex =
    /\[download\]\s+([\d.]+)%\s+of\s+([^\s]+).*?at\s+([^\s]+).*?ETA\s+([^\s]+)/;

  const match = line.match(progressRegex);

  if (!match) return null;

  return {
    percent: Number(match[1]),
    totalSize: match[2],
    speed: match[3],
    eta: match[4]
  };
}
```

---

## 10.4 状态解析函数

```js
function parseTaskStage(line) {
  if (line.includes("[Merger]")) return "merging";
  if (line.includes("[ExtractAudio]")) return "postprocessing";
  if (line.includes("[Metadata]")) return "postprocessing";
  if (line.includes("[EmbedSubtitle]")) return "postprocessing";
  if (line.includes("[download]")) return "downloading";
  return null;
}
```

---

## 10.5 注意事项

1. yt-dlp 输出可能在 stdout，也可能在 stderr。
2. 必须同时监听 stdout 和 stderr。
3. 必须添加 `--newline`，方便逐行解析。
4. 不同站点输出可能略有差异。
5. 解析失败时仍然要保留 raw log。

## 14. 错误处理方案

---

## 14.1 错误分类

| 类型 | 示例 | 处理 |
|---|---|---|
| URL 错误 | Unsupported URL | 提示链接不支持 |
| 网络错误 | timeout / unable to download webpage | 提示检查网络或代理 |
| 权限错误 | 403 / Sign in | 提示使用 Cookie 或代理 |
| 格式错误 | format not available | 提示更换格式 |
| FFmpeg 错误 | ffmpeg not found | 提示检查 FFmpeg |
| 文件错误 | permission denied | 提示检查保存目录权限 |
| 用户取消 | process killed | 标记 canceled |
| 未知错误 | unknown error | 显示原始日志摘要 |

---

## 14.2 错误转换函数

```js
function normalizeYtdlpError(errorText) {
  if (errorText.includes("Unsupported URL")) {
    return "当前链接暂不支持或链接格式错误。";
  }

  if (errorText.includes("Video unavailable")) {
    return "视频不可用，可能已删除、私密或地区限制。";
  }

  if (errorText.includes("HTTP Error 403")) {
    return "访问被拒绝，可能需要代理或 Cookie。";
  }

  if (errorText.includes("Sign in")) {
    return "该内容可能需要登录，请导入 Cookie 后重试。";
  }

  if (errorText.toLowerCase().includes("ffmpeg")) {
    return "FFmpeg 相关错误，请检查 FFmpeg 是否存在。";
  }

  if (errorText.includes("Requested format is not available")) {
    return "当前清晰度或格式不可用，请选择其他格式。";
  }

  return errorText || "下载失败，请查看日志。";
}
```

## 21. 风险分析

---

## 21.1 yt-dlp 站点规则变化

风险：

视频网站规则变化导致解析失败。

解决：

1. 提供更新 yt-dlp 功能。
2. 显示清晰错误。
3. 提供日志导出。
4. 不承诺所有网站永久可用。

---

## 21.2 Cookie 使用风险

风险：

Cookie 泄露可能导致账号安全问题。

解决：

1. Cookie 只本地使用。
2. 不上传。
3. 不写入日志。
4. UI 明确提示风险。

---

## 21.3 杀毒误报

风险：

内置 exe 和调用子进程可能被部分杀毒软件误报。

解决：

1. 使用官方二进制来源。
2. 后续使用代码签名。
3. 不执行远程脚本。
4. 不隐藏进程行为。

---

## 21.4 路径和权限问题

风险：

Windows 路径复杂导致下载失败。

解决：

1. 使用 spawn 参数数组。
2. 使用 path.join。
3. 保存前检查目录权限。
4. 提供错误提示。

---

## 21.5 UI 功能过多导致复杂

风险：

一次性开发全部功能导致项目失控。

解决：

1. 严格按 V1、V2、V3 分阶段。
2. V1 只做核心链路。
3. 每个阶段独立验收。
4. 不在 V1 做过多高级功能。

## 23.1 ytdlpService 基础实现

```js
const { spawn } = require("child_process");

function runYtdlp(ytdlpPath, args, onData) {
  return new Promise((resolve, reject) => {
    const child = spawn(ytdlpPath, args, {
      windowsHide: true
    });

    let stdout = "";
    let stderr = "";

    child.stdout.on("data", (data) => {
      const text = data.toString("utf8");
      stdout += text;
      if (onData) onData(text);
    });

    child.stderr.on("data", (data) => {
      const text = data.toString("utf8");
      stderr += text;
      if (onData) onData(text);
    });

    child.on("error", (err) => {
      reject(err);
    });

    child.on("close", (code) => {
      if (code === 0) {
        resolve({ stdout, stderr });
      } else {
        reject(new Error(stderr || stdout || `yt-dlp exited with code ${code}`));
      }
    });
  });
}
```

## 23.3 下载实现

```js
function startDownloadTask({ ytdlpPath, args, task, onProgress, onLog }) {
  const child = spawn(ytdlpPath, args, {
    windowsHide: true
  });

  task.process = child;
  task.processId = child.pid;

  child.stdout.on("data", (data) => {
    handleOutput(data.toString("utf8"));
  });

  child.stderr.on("data", (data) => {
    handleOutput(data.toString("utf8"));
  });

  function handleOutput(text) {
    onLog?.(text);

    const lines = text.split(/\r?\n/);

    for (const line of lines) {
      const progress = parseProgressLine(line);
      const stage = parseTaskStage(line);

      if (progress) {
        onProgress({
          ...progress,
          status: "downloading"
        });
      }

      if (stage) {
        onProgress({
          status: stage
        });
      }
    }
  }

  return child;
}
```

## 24. 结论

本技术方案的核心是：

```text
不要重写下载器，
而是把 yt-dlp 的强大能力封装成安全、稳定、清晰的桌面 UI。
```

最关键的工程点是：

1. Electron 主进程和 React 前端必须分层清晰。
2. preload.js 只暴露安全 API。
3. commandBuilder 统一生成 yt-dlp 参数。
4. ytdlpService 统一调用 yt-dlp。
5. taskQueue 统一管理任务状态。
6. configStore / historyStore / logService 统一管理本地数据。
7. 所有命令调用必须使用 spawn 参数数组。
8. 所有高级功能分阶段开发，先保证 V1 核心链路稳定。

建议开发顺序：

```text
框架 → IPC → 解析 → 下载 → 进度 → 队列 → 配置 → 历史 → 高级功能 → 打包
```

只要按照这个方案推进，项目可以较稳地落地为一个完整的 Windows 桌面 yt-dlp 下载管理器。
